# Рекомендации: архив и порядок в D:\Projects

Папка **`D:\Projects\_archive`** предназначена для переноса устаревших или дублирующих материалов. В Git она не попадает (см. `.gitignore`).

---

## Что можно перенести в `_archive`

| Элемент | Рекомендация |
|--------|---------------|
| **Cursor.lnk** | Ярлык; можно перенести в `_archive` или удалить. |
| **AI_ASSISTANT_ACCESS_GUIDE.md** | Если инструкция дублирует другие гайды или устарела — в `_archive` или актуализировать в `docs/`. |
| **HampfreeBlog/** | Если это старая копия блога (до Astro / marketlab-academy) — перенести в `_archive` после проверки, что всё нужное есть в текущем репо. |
| **Veles Blog/** | Отдельный большой проект (5k+ файлов). Если не ведётся — в `_archive` или отдельный диск/репо. |
| **AutomationHub/** | Набор подпроектов (ContentBot, MarketLabCtaBot, n8n и т.д.). Готовые/замороженные части можно вынести в `_archive` или оставить, если активно используете. |
| **ai-assistants/** | Документация по ботам/DeepSeek. В `_archive`, если не актуально. |
| **cursor-talk-to-figma-mcp/** | MCP-плагин. В `_archive`, только если больше не используете. |
| **marketlab-content/** | Контент-репо (статьи, платформы). Не архивировать, если контент живой; при смене структуры — обновить пути в скриптах. |

---

## Что лучше не архивировать

- **src/, public/, scripts/, docs/, package.json** — ядро marketlab-academy, не трогать.
- **HampfreeBlog-Private/** — приватный репо; не перемещать в общий `_archive` без необходимости.
- **marketlab-content/** — пока контент актуален, оставить на месте.

---

## Порядок действий

1. Создать папку `D:\Projects\_archive` (если ещё нет).
2. Переносить по одному пункту из таблицы, после каждого — проверить, что сборка и скрипты работают.
3. Актуализировать пути в скриптах (например, `check-internal-links.mjs` уже использует `process.cwd()`).
4. Обновить этот файл, когда что-то перенесено или решение изменено.
